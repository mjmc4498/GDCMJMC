📋 Descripción del Módulo

La Integración de Datos comprende los procesos, tecnologías y arquitecturas que consolidan, transforman y sincronizan datos provenientes de múltiples fuentes heterogéneas para entregar información consistente y oportuna a los consumidores de datos. Es el puente que conecta los sistemas fuente con los sistemas analíticos y operacionales.

📘 Relación con DAMA-DMBOK

En el DAMA-DMBOK, la Integración e Interoperabilidad de Datos cubre los patrones ETL (Extract, Transform, Load), ELT, Change Data Capture (CDC), virtualización de datos, APIs de datos y servicios de datos. Define las actividades para diseñar, implementar y operar flujos de datos que garanticen la calidad y consistencia de la información integrada.

⚖️ Relación con Gobierno Corporativo

Desde el gobierno corporativo, la integración de datos tiene implicaciones estratégicas directas: los reportes ejecutivos, los dashboards del directorio y la inteligencia de negocio dependen de la calidad y confiabilidad de los procesos de integración. Una integración deficiente genera reportes incorrectos que conducen a decisiones de negocio basadas en información errónea.

🔒 Cumplimiento — Ley N.° 29733

Cuando los flujos de integración transfieren datos personales entre sistemas, aplica plenamente la Ley 29733. El artículo 15 regula las condiciones para el flujo de datos entre entidades del mismo grupo económico. El artículo 35A regula las transferencias internacionales de datos personales, exigiendo garantías equivalentes de protección en el país receptor.

🎯 Objetivos de Aprendizaje

  • 🎯 Diseñar e implementar pipelines de datos usando patrones ETL, ELT y CDC del marco DAMA-DMBOK.
  • 🎯 Comparar y seleccionar entre arquitecturas de integración: batch, micro-batch y streaming.
  • 🎯 Implementar controles de calidad de datos en los procesos de integración.
  • 🎯 Diseñar arquitecturas de integración resilientes con manejo de errores y reintentos.
  • 🎯 Cumplir la Ley 29733 en flujos de datos personales entre sistemas y con el exterior.

🔑 Conceptos Clave

  • 🔑 ETL (Extract, Transform, Load): Proceso de extracción de datos de fuentes, transformación según reglas de negocio y carga en destino.
  • 🔑 ELT (Extract, Load, Transform): Variante moderna donde los datos se cargan primero en el destino y se transforman allí, aprovechando el poder de cómputo del data warehouse.
  • 🔑 CDC (Change Data Capture): Técnica que captura solo los cambios incrementales en las fuentes para una integración eficiente y casi en tiempo real.
  • 🔑 Data Virtualization: Capa de abstracción que presenta datos de múltiples fuentes sin moverlos físicamente.
  • 🔑 API de Datos: Interfaz programática que expone datos de forma segura y estandarizada para consumo interno y externo.

⚙️ Explicación Técnica Detallada

Los patrones de Integración e Interoperabilidad se clasifican según su latencia y volumen:

  • 📦 1. Batch (Por Lotes): Integración programada de grandes volúmenes con baja necesidad de actualización inmediata (ETL tradicional).
  • 🔄 2. Micro-batch: Integración frecuente en pequeños lotes de tiempo (ej. Apache Spark Structured Streaming).
  • ⚡ 3. Streaming (Tiempo Real): Procesamiento evento a evento continuo (ej. Apache Kafka, AWS Kinesis).
  • 🌐 4. Virtualización: Acceso federado y en tiempo real sin mover físicamente los datos.

En ecosistemas modernos, la Arquitectura Medallion (Bronze ➔ Silver ➔ Gold) organiza las capas de transformación en Data Lakehouses, garantizando trazabilidad y calidad de datos progresiva.

🏢 Ejemplos en Organizaciones

La empresa de transporte aéreo LATAM Airlines implementó un pipeline de integración en tiempo real con Apache Kafka para sincronizar datos de reservas entre 9 sistemas (GDS, check-in, equipaje, catering) en 14 países. El tiempo de sincronización pasó de 30 minutos a 3 segundos, eliminando inconsistencias que generaban reclamos de pasajeros y multas regulatorias.

✅ Buenas Prácticas Internacionales

  • ✔️ Implementar idempotencia en todos los pipelines de integración para garantizar resultados consistentes ante reejecutciones.
  • ✔️ Diseñar siempre un mecanismo de gestión de errores con alertas y registro de trazabilidad.
  • ✔️ Usar CDC en lugar de full-load para integraciones frecuentes, reduciendo carga en sistemas fuente.
  • ✔️ Documentar el lineaje de datos de cada pipeline para auditoría y diagnóstico.
  • ✔️ Cifrar los datos en tránsito con TLS 1.3 en todos los flujos que incluyan datos personales.

⚠️ Errores Comunes

  • ❌ Diseñar pipelines de integración sin gestión de errores, generando pérdida silenciosa de datos.
  • ❌ No documentar las transformaciones aplicadas, dificultando el diagnóstico de problemas.
  • ❌ Integrar datos personales entre sistemas sin verificar el cumplimiento de la Ley 29733.
  • ❌ Usar consultas SQL directas a bases de datos de producción para integración, generando carga no planificada.
  • ❌ No implementar monitoreo de latencia y calidad en los pipelines, detectando problemas solo cuando usuarios se quejan.

💼 Casos de Uso Reales

Rímac Seguros en Perú integró datos de 5 sistemas core (pólizas, siniestros, clientes, pagos, agentes) usando MuleSoft, implementando una capa de API Management que redujo el tiempo de desarrollo de nuevas integraciones de 3 meses a 2 semanas. El catálogo de APIs incluye controles de throttling y auditoría de acceso para cumplir con la Ley 29733.

🛠️ Herramientas y Tecnologías

  • Apache Kafka: Plataforma de streaming distribuido para integración en tiempo real de alta disponibilidad.
  • Apache Airflow: Orquestador de pipelines de datos batch y micro-batch con interfaz visual de monitoreo.
  • dbt (data build tool): Herramienta de transformación y documentación de modelos en el patrón ELT moderno.
  • MuleSoft / Azure Integration Services: Plataformas de integración empresarial con gestión de APIs y conectores listos para usar.
  • AWS Glue / Azure Data Factory: Servicios de integración de datos en la nube con catálogo de datos integrado.

✏️ Ejercicio Práctico

Escenario: PlazaVea Express opera 80 tiendas con sistemas POS Oracle, un CRM con datos de 2.5 millones de clientes con tarjeta de fidelización, y un Data Warehouse en Snowflake. Los reportes de ventas por categoría deben estar listos a las 6 AM del día siguiente. Además, el área de Marketing requiere datos de comportamiento de compra en cuasi-tiempo-real para activar promociones personalizadas en la app móvil. Actualmente, los datos llegan con 48 horas de retraso y un 12% de registros se pierden silenciosamente.

Entregables:

  • 1. Diagrama de arquitectura del pipeline con patrón Medallion (Bronze → Silver → Gold): fuentes, ingestión, staging, transformación y capa de consumo. Usa notación de Enterprise Integration Patterns donde aplique.
  • 2. Estrategia híbrida de integración: CDC (Debezium/Kafka Connect) para actualización de inventario en tiempo real + ETL batch nocturno (Airflow + dbt) para el DW analítico. Justifica por qué cada patrón es adecuado para cada caso de uso.
  • 3. Reglas de transformación documentadas: estandarización de códigos de producto (SKU unificado), normalización de nombres de sucursal, conversión de zonas horarias, deduplicación de transacciones y cálculo de métricas derivadas.
  • 4. Plan de gestión de errores y observabilidad: Dead Letter Queue para registros fallidos, alertas automáticas (Slack/PagerDuty), reintentos con backoff exponencial, dashboard de latencia y completitud, y SLA de 99.5% de entrega.
  • 5. Controles de privacidad en el pipeline: anonimización de datos de clientes en la capa Silver, tokenización de DNI, cifrado TLS 1.3 en tránsito, logs de auditoría de acceso a datos personales. Cumplimiento Ley 29733 art. 15 (flujo entre entidades del grupo).

Formato de entrega: Diagrama de arquitectura + documento de reglas de transformación + plan de errores. Marcos sugeridos: EAI Patterns, Medallion Architecture, Apache Integration Patterns.

💡 Recomendaciones de Implementación

  • 📌 Diseñar los pipelines con el patrón Medallion (Bronze/Silver/Gold) para garantizar calidad progresiva.
  • 📌 Implementar un catálogo de APIs de datos para estandarizar el consumo y agregar control de acceso.
  • 📌 Adoptar herramientas de orquestación (Airflow, Prefect) desde el inicio para gestionar dependencias.
  • 📌 Incluir métricas de calidad de datos (completitud, exactitud, oportunidad) en los pipelines.
  • 📌 Documentar cada pipeline con su propósito, fuentes, transformaciones y SLA de actualización.

📌 Resumen del Módulo

La Integración de Datos es el tejido conectivo del ecosistema de información empresarial. El DAMA-DMBOK la define como un dominio crítico que requiere tanto rigor técnico (arquitecturas resilientes, CDC, streaming) como gobierno (lineaje, calidad, cumplimiento). En el contexto peruano, todo flujo de datos personales debe ser evaluado bajo la Ley 29733, especialmente las transferencias entre sistemas y con terceros o el extranjero.
